# CS-Scout 2.0 — 多回合叠加回放 设计文档

**日期**: 2026-06-24
**状态**: 已批准设计，待写实现计划

## 目标

赛前侦查工具。用户输入 **5 名对手用户名 + 1 张地图**，系统自动从 5E 平台抓取每人在该图的 demo，边下边解析，为每名选手生成 **CT/T 两方 × Pistol/Full/Eco 三局型 = 6 个 2D 加速回放动图**。同一 (方位, 局型) 下的所有回合在同一块 canvas 上叠加，沿统一时间轴同时播放，呈现选手的开局站位/进攻习惯，并叠加投掷物的飞行弧线、落点与遮挡/燃烧范围。参考 2026 科隆 Major 赛前职业选手行动图，天梯版。

## 与 1.0 的关系

1.0 产出的是按局型分的**静态密度热图 PNG**（matplotlib），只分析 CT 方、写死 Mirage、含 zone 占比统计。2.0 是方向性重写：

| 维度 | 1.0 | 2.0 |
|------|-----|-----|
| 核心产物 | 静态热图 PNG | 前端 canvas 实时 9s 循环回放动图 |
| 方位 | 仅 CT | CT + T |
| 地图 | 写死 Mirage | 当前赛季 7 图 |
| 投掷物 | 无 | 飞行弧线 + 落点 + 遮挡/燃烧范围 |
| 局型 | Pistol/Full/Force/Eco（4 类） | Pistol/Full/Eco（3 类），CT/T 同模型 |
| 渲染端 | 服务器 matplotlib | 前端 canvas（服务器只吐 JSON） |
| 附加统计 | K/D、AWP 率、zone 占比 | K/D、AWP 率（去 zone） |

**复用约 55%**：5E 抓取链路（`api_client.py`）、双线程边下边解析流水线、demo 去重索引、磁盘清理、Flask 服务 + 状态轮询、局型分类逻辑、坐标 transform 概念。

**移除**：matplotlib、ffmpeg、zone 多边形系统、热图生成。

## 架构

```
浏览器 UI (index.html)
   │  POST /api/analyze  {usernames[5], map, max_demos, key}
   ▼
web_server.py (Flask, 端口 5000)
   │  后台线程
   ▼
pipeline.run()  —— 解析-only，产出 JSON
   ├─[下载线程] search_player → get_demos_by_domain(domain, map)
   │            → get_steamid_for_player → download_and_extract（去重索引）
   └─[主线程]   parse_demo: 坐标(CT+T, 0–45s, 降采样) + grenades
   │            + 局型分类(3类双向) + K/D/AWP
   ▼
output/player_{domain}.json  +  output/analysis_summary.json
   ▼
浏览器 canvas 引擎: 加载 maps/{map}/radar.png + transform → 实时回放
```

## 组件设计

### 1. 地图层 `server/data/maps/{map}/`

每张当前赛季现役图一个目录（实际清单以扫描时的 active duty 为准，候选池：de_mirage / de_inferno / de_nuke / de_overpass / de_ancient / de_anubis / de_dust2 / de_train，取其中 7 张现役），含：

- `radar.png` — 雷达底图
- transform — `{pos_x, pos_y, scale}`，game→pixel 换算：
  `px = (gx - pos_x) / scale`，`py = (pos_y - gy) / scale`（Y 轴反向）

**数据来源**：`awpy` 2.x 的 `awpy.data.map_data.MAP_DATA` 提供全部现役图的 transform，radar 图随 `awpy` 地图资源下载（`awpy get maps` 一次性拉取到 `~/.awpy/maps`）。安装脚本把它们复制/链接进 `data/maps/`。transform 通过 API 随玩家 JSON 一并下发，供前端做 game→canvas 像素换算。

> **实现期需验证**：awpy 2.0.2 的 `MAP_DATA` 在 `awpy get maps` 后填充，确认 7 张现役图的 `pos_x/pos_y/scale` 与实际 radar.png 分辨率匹配；若某图缺失则回退到社区 SimpleRadar + 手工标定（沿用 1.0 的 map_config.json 格式）。

### 2. 抓取层 `server/api_client.py`

沿用 1.0，唯一改动是泛化地图过滤：

- `get_demos_by_domain(domain, map_name, count)` —— 由 1.0 的 `get_mirage_demos_by_domain` 改写，把写死的 `"de_mirage"` 换成 `map_name` 参数。保留 match_type=9 优先、分页、`seen_codes` 去重逻辑。
- `search_player` / `get_steamid_for_player` / `download_demo` 不变。

### 3. 解析层 `server/pipeline.py`

保留：双线程下载/解析流水线（`run_by_usernames` 改名 `run`）、demo 去重索引、`cleanup_demos`、回合编号去重偏移。

**移除**：`generate_heatmap`、`_render_rtype_ax`、`generate_zone_stats`、`load_zones`/`load_weights`、所有 matplotlib/scipy/shapely 依赖。

**局型分类**（改 `classify_rounds`）：3 类，CT 和 T 用**同一套**模型——每半场第一回合 = Pistol，其余按队伍均 `current_equip_value` 分 Full（≥ 阈值）/ Eco。CT 和 T 都要分类（去掉只取 CT 的过滤）。

**坐标解析**（改 `parse_demo`）：
- 窗口：每回合 `freeze_end` → `freeze_end + 45s`（不再依赖首杀）。
- 采样：降采样到约 8 Hz（每 ~8 ticks 取一点），减小 JSON 体积。
- 双方：CT 和 T 回合都采，按 side 标注。
- 仅追踪目标 steamid 自己的坐标（单选手，多回合叠加）。
- 输出每回合一条 `path: [[t, x, y], ...]`，t 为相对 freeze_end 的秒数。

**投掷物解析**（新增 `parse_grenades_for_rounds`）：
- `parser.parse_grenades()` 取 `grenade_type` 含 `Projectile` 的行（裸库存类型 x/y 为 NaN，剔除）。
- 仅取目标 steamid 投掷的、落在某回合 0–45s 窗口内的。
- 每枚算出：`type`（smoke/flash/he/molotov/incendiary/decoy）、`throw_t`、飞行 `arc: [[t,x,y],...]`、`land`（最后一个有效点）、`land_t`、`expire_t`（烟 ~18s、火 ~7s 持续；闪/雷/诱饵 land_t 后即消失）。

**战斗统计**：保留 `parse_combat_stats` / `aggregate_combat_stats`（K/D 全局记分板 + AWP 率 CT 侧）。

### 4. 输出 JSON

每选手一个 `output/player_{domain}.json`：

```json
{
  "username": "...", "domain": "...", "steamid": "...",
  "map": "de_mirage",
  "transform": {"pos_x": -3230, "pos_y": 1713, "scale": 5.0},
  "radar": "/maps/de_mirage/radar.png",
  "combat_stats": {"kd": 1.23, "awp_rate": 45.0},
  "demos_found": 6, "round_count": 47,
  "rounds": [
    {
      "side": "CT", "rtype": "Full", "round_id": 3,
      "path": [[0.0, -1200, 340], [0.13, -1190, 352], "..."],
      "grenades": [
        {"type": "smoke", "throw_t": 8.1, "land_t": 9.4,
         "arc": [[8.1,-1100,300], "..."], "land": [-900, 250],
         "expire_t": 27.4}
      ]
    }
  ]
}
```

`output/analysis_summary.json`：汇总 `{map, max_demos, failed[], results[{username, domain, player_json, combat_stats, round_count, demos_found}]}`。

### 5. 前端 canvas 引擎 `server/templates/index.html`（+ 静态 JS）

- 输入区：5 个用户名框 + **地图下拉** + demo 深度 + Execute。状态轮询（沿用 1.0 的 `/api/status` 2s 轮询）。
- 每选手一张卡：
  - 卡头：选手名 + K/D + AWP 率。
  - **CT/T tab 切换**：默认 CT。
  - 当前 tab 下三块 canvas：Pistol / Full / Eco。每块加载 radar 底图，叠加该 (side, rtype) 下所有回合。
- **canvas 回放引擎**（核心新模块，独立 JS）：
  - 9s 自动循环。45s → 9s（5× 压缩，倍速常量可调；预留按需压缩死时间的优化点）。
  - 每帧按当前虚拟时间 t：
    - 每个开启回合画玩家点（纯点，**无拖尾**）。
    - 投掷物：`throw_t ≤ t < land_t` 时按 arc 画**飞行弧线**（飞到当前 t 的位置，**无渐隐拖尾**）；`t ≥ land_t` 画落点标记；smoke/molotov 在 `land_t ≤ t < expire_t` 画遮挡/燃烧范围圈。
  - 轻量交互：**逐回合开关**复选框、hover 高亮某条轨迹。**无时间轴拖动**。
  - game→canvas 换算用 JSON 里的 `transform`。

## 局型与窗口参数（config）

- `WINDOW_S = 45`（回合窗口秒数）
- `SAMPLE_HZ ≈ 8`（坐标降采样）
- `PLAYBACK_S = 9`（动图时长）→ 加速比 5×
- `EQ_FULL_BUY`（Full 阈值，沿用 1.0 config）；< 阈值即 Eco；半场首回合 = Pistol
- 烟雾持续 `SMOKE_DUR ≈ 18s`、火 `FIRE_DUR ≈ 7s`

## 已知权衡与限制

- 产物依赖服务器在线，不能当独立文件转发（已与用户确认接受）。
- canvas 回放引擎是新增前端工作量，但边界清晰、可独立测试。
- 沿用 1.0 的数据限制：无该图天梯局的选手抓不到 demo（报失败）；`get_steamid_for_player` 按用户名匹配，改名则失败；K/D 为全局双向。
- 9s 固定加速为初版默认；「压缩死时间/变速」列为后续优化，不在首版范围。

## 非目标（YAGNI）

- 不做服务器端视频/GIF 导出（前端 canvas 替代）。
- 不做整队 5 人配合图（仅单选手多回合叠加）。
- 不做 zone 占比统计。
- 不做时间轴拖动、不做 Force Buy 第四局型。
